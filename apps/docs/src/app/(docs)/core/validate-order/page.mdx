# Validate Order

`client.core.validateOrder()` confirms a payment by querying SSLCommerz's validation server with the `val_id` received from either the IPN callback or the redirect URL. This is the definitive confirmation step — never update your database based on IPN or redirect data alone.

**Underlying endpoint:** `GET validator/api/validationserverAPI.php`

```ts
const validation = await client.core.validateOrder(input, options?);
```

## When to call it

Call `validateOrder` in **two places**:

1. **In your IPN handler** — after receiving an IPN POST with `status: "VALID"`, validate the `val_id` before marking the order as paid.
2. **In your success redirect handler** — after the customer is redirected back to your `success_url`, validate the `val_id` from the POST data as a fallback (IPN can sometimes be delayed).

## Input

<Properties>
  <Property name="val_id" type="string" required>
    The validation ID from the IPN payload (`ipn.val_id`) or from the
    success/fail redirect POST body. Max 50 characters.
  </Property>
  <Property name="format" type='"json" | "xml"' optional>
    Response format requested from SSLCommerz. Defaults to `"json"`.
  </Property>
  <Property name="v" type="string" optional>
    Reserved field accepted by the endpoint.
  </Property>
</Properties>

`store_id` and `store_passwd` are automatically injected from the client — do not include them.

## Example

```ts
// app/api/ipn/route.ts
import { sslcommerz } from "@/lib/sslcommerz";

export async function POST(request: Request) {
  const form = await request.formData();
  const valId = form.get("val_id") as string;

  const validation = await sslcommerz.core.validateOrder({ val_id: valId });

  // 1. Check the validation status
  if (validation.status !== "VALID" && validation.status !== "VALIDATED") {
    return new Response("Payment not valid", { status: 400 });
  }

  // 2. Cross-check tran_id against your database
  const order = await db.orders.findByTranId(validation.tran_id);
  if (!order) {
    return new Response("Order not found", { status: 404 });
  }

  // 3. Cross-check the amount
  if (Number(validation.amount) !== order.expectedAmount) {
    console.error("Amount mismatch — possible tampering detected");
    return new Response("Amount mismatch", { status: 400 });
  }

  // 4. Check risk level
  if (validation.risk_level === "1") {
    await db.orders.flagAsRisky(order.id);
    return new Response("OK");
  }

  // 5. All clear — fulfil the order
  await db.orders.markAsPaid(order.id, {
    valId: validation.val_id,
    bankTranId: validation.bank_tran_id,
  });

  return new Response("OK");
}
```

## Success redirect handler

```ts
// app/payment/success/route.ts
import { NextResponse } from "next/server";
import { sslcommerz } from "@/lib/sslcommerz";

// SSLCommerz POSTs form data to success_url
export async function POST(request: Request) {
  const form = await request.formData();
  const valId = form.get("val_id") as string;

  const validation = await sslcommerz.core.validateOrder({ val_id: valId });

  if (validation.status !== "VALID" && validation.status !== "VALIDATED") {
    return NextResponse.redirect(new URL("/payment/failed", request.url));
  }

  // Order verified — redirect to confirmation page
  return NextResponse.redirect(
    new URL(`/orders/${validation.tran_id}`, request.url),
  );
}
```

## Security checklist

Before updating your database or fulfilling an order:

1. **`status` must be `"VALID"` or `"VALIDATED"`** — any other value means the payment is not confirmed. Reject it.
2. **Cross-check `tran_id`** — it must match an order in your database. Reject if not found.
3. **Cross-check `amount`** — it must match the expected order amount. A mismatch indicates possible tampering.
4. **Cross-check `currency`** — it must match what you expected. Do not fulfil if the currency differs.
5. **Handle `risk_level = "1"`** — do not automatically fulfil. Hold the order and verify the customer manually.
6. **Handle `"VALIDATED"` status** — this means the `val_id` was already validated once. Your idempotency logic should prevent double-fulfilment regardless of this status.

<Note>
  Store `bank_tran_id` from the response — you will need it if you ever need to
  [initiate a refund](/core/refund) for this transaction.
</Note>
