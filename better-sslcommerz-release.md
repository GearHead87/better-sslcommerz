# Better SSLCommerz Release Plan

## Goal

Publish these three packages with automated versioning and changelog generation through GitHub Actions:

- `@better-sslcommerz/validators`
- `@better-sslcommerz/sdk`
- `@better-sslcommerz/convex`

## Current blockers to fix first

1. `packages/validators/package.json` is `private: true` and currently exports TypeScript source (`./src/index.ts`) instead of publish-ready JavaScript.
2. `packages/convex/package.json` is `private: true`, so npm publish is blocked.
3. There is no release system yet (`.changeset` missing, no release workflow).
4. Changelog files are not present for package-level release notes.

## Recommended release model

Use **Changesets + GitHub Actions** with a **lockstep version** for these three packages while the ecosystem is still early.

Why lockstep now:

- The packages are tightly related (`validators -> sdk -> convex`).
- It prevents dependency drift during early releases.
- It keeps release communication simple.

## Version strategy

- Initial public release: **`0.1.0`** for all three packages.
- Ongoing bumps:
  - `patch`: bug fixes, docs, internal improvements with no API behavior change.
  - `minor`: new APIs/features.
  - `major`: breaking API changes.

Recommended baseline matrix:

| Package | Initial version |
| --- | --- |
| `@better-sslcommerz/validators` | `0.1.0` |
| `@better-sslcommerz/sdk` | `0.1.0` |
| `@better-sslcommerz/convex` | `0.1.0` |

## GitHub Actions release flow

### 1) PR flow (developer side)

1. Developer changes one or more of the three packages.
2. Developer runs `pnpm changeset` and writes a short release note.
3. PR is merged to `main` after CI passes.

### 2) Main branch flow (automation)

`release.yml` runs on push to `main` (and `workflow_dispatch`):

1. Checkout + setup Node/pnpm.
2. Install deps.
3. Run release quality checks (recommended):
   - `pnpm lint`
   - `pnpm typecheck`
   - `pnpm --filter @better-sslcommerz/validators build`
   - `pnpm --filter @better-sslcommerz/sdk build`
   - `pnpm --filter @better-sslcommerz/convex build`
   - `pnpm --filter @better-sslcommerz/convex test`
4. Run `changesets/action`:
   - If unpublished changesets exist: open/update a Release PR with version bumps + changelog updates.
   - If release PR changes are already on `main`: publish packages to npm.
5. Create GitHub Releases from changelog entries (`createGithubReleases: true`).

### 3) Publish order and dependency safety

Changesets handles topological publishing for changed packages; expected order is:

1. `@better-sslcommerz/validators`
2. `@better-sslcommerz/sdk`
3. `@better-sslcommerz/convex`

## Changelog policy

- Use package-level changelogs generated by Changesets.
- Every functional PR touching publishable packages includes one changeset file.
- Changeset summary style:
  - first line: user-facing change
  - include migration note when breaking
  - use `BREAKING:` prefix for breaking behavior

Example changeset:

```md
---
"@better-sslcommerz/validators": minor
"@better-sslcommerz/sdk": minor
"@better-sslcommerz/convex": minor
---

Add first stable public release of validators, SDK, and Convex component with typed runtime exports and release automation.
```

## Secrets and permissions required

GitHub repository secrets:

- `NPM_TOKEN` (npm automation token with publish rights)

Workflow permissions in `release.yml`:

- `contents: write`
- `pull-requests: write`
- `id-token: write` (recommended for npm provenance)
