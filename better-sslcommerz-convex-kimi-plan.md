**Critical observations:**

- **`files: ["dist", "src"]`** — BOTH compiled output AND original source are published. The `src/` is needed because `./test` and `./convex.config.js` point into `src/` or need source for component registration.
- **`types` field at top-level** — `"types": "./dist/client/index.d.ts"` for legacy TypeScript resolution.
- **`module` field at top-level** — `"module": "./dist/client/index.js"` for bundlers that check this.
- **Build command**: `tsc --project ./tsconfig.build.json` — plain TypeScript, no bundler/tsdown. This is intentional because Convex components use TypeScript's native ESM output for tree-shaking.
- **Build codegen**: `npx convex codegen --component-dir ./src/component` — must be run BEFORE the main build so `_generated/` is populated.
- The `convex.json` points `"functions"` to `"example/convex"` (the example app's convex dir), NOT to `src/component`. The component itself is registered via `convex.config.ts`.

### 2.6 `src/client/index.ts` — The `Polar` Class (demo-project/polar/src/client/index.ts:1-522)

The main class is significantly more sophisticated than Stripe's:

```typescript
class Polar<DataModel, Products extends Record<string, string>> {
  constructor(
    public component: ComponentApi,
    private config: {
      products?: Products;
      getUserInfo: (ctx: RunQueryCtx) => Promise<{ userId: string; email: string }>;
      organizationToken?: string;
      webhookSecret?: string;
      server?: "sandbox" | "production";
    }
  )

  // Data access
  getCustomerByUserId(ctx, userId)
  async syncProducts(ctx)
  async createCheckoutSession(ctx, { productIds, userId, email, origin, successUrl, ... })
  async createCustomerPortalSession(ctx, { userId })
  listProducts(ctx, { includeArchived? })
  async getCurrentSubscription(ctx, { userId })    // Returns subscription + productKey + product
  listAllUserSubscriptions(ctx, { userId })
  getProduct(ctx, { productId })
  async changeSubscription(ctx, { productId })
  async cancelSubscription(ctx, { revokeImmediately? })

  // Returns Convex functions to register in user's convex/ folder
  api() → {
    changeCurrentSubscription: actionGeneric(...)
    cancelCurrentSubscription: actionGeneric(...)
    getConfiguredProducts: queryGeneric(...)
    listAllProducts: queryGeneric(...)
    listAllSubscriptions: queryGeneric(...)
    generateCheckoutLink: actionGeneric(...)   // Main checkout flow
    generateCustomerPortalUrl: actionGeneric(...)
  }

  // HTTP webhook handler registration
  registerRoutes(http: HttpRouter, {
    path?: string;
    events?: WebhookEventHandlers;   // Typesafe per-event handlers
    onSubscriptionCreated?: ...;     // deprecated
    onSubscriptionUpdated?: ...;     // deprecated
    onProductCreated?: ...;          // deprecated
    onProductUpdated?: ...;          // deprecated
  })
}
```

**Key pattern:** `api()` method returns an object of Convex functions that the user registers in their own `convex/` folder:

```typescript
// In user's convex/checkout.ts
export const { generateCheckoutLink, generateCustomerPortalUrl, ... } = polar.api();
```

### 2.8 Key Architectural Difference Between Stripe and Polar

| Aspect                    | Stripe                              | Polar                                   |
| ------------------------- | ----------------------------------- | --------------------------------------- |
| Component functions split | public.ts + private.ts              | Single lib.ts                           |
| React components          | Empty stub                          | Full CustomerPortalLink + CheckoutLink  |
| Client class pattern      | Class instance                      | Class instance with `api()` method      |
| Product catalog syncing   | No                                  | Yes — `syncProducts()` action           |
| Webhook event types       | Stripe-specific events              | Polar webhook events (standardwebhooks) |
| `getUserInfo` config      | No (uses stripeCustomerId directly) | Yes — injected callback                 |
| Trial period support      | Via Stripe prices                   | Native in checkout API                  |
| Server mode               | N/A                                 | `"sandbox"` or `"production"`           |

### 3.5 Build System (packages/better-sslcommerz/tsdown.config.ts:1-10)

Uses `tsdown` (a bundler, not plain tsc) to produce both ESM and CJS:

```typescript
export default defineConfig({
  entry: { index: "src/index.ts" },
  format: ["esm", "cjs"],
  dts: false, // Declaration types generated by separate tsc step
});
```

The `build` script: `tsdown && tsc -p tsconfig.build.json`

This is fundamentally different from stripe/polar which use only `tsc`. The Convex component **must** use plain `tsc` (not a bundler) because:

1. Convex's `convex codegen` generates import paths expecting `.js` extensions in ESM style
2. Convex's runtime consumes the source TypeScript files of the component directly during development
3. Bundling breaks the module graph that Convex's component system expects

## 5. Package Reuse Analysis — Can We Reuse the Current Package?

This is the critical question. Two tracks:

### Track A: Reuse `better-sslcommerz` as an npm Dependency

**Feasibility: YES — RECOMMENDED**

The Convex component `@convex-dev/better-sslcommerz` needs to make HTTP calls to SSLCommerz's API from within Convex actions. The existing `better-sslcommerz` package already provides:

1. **`createSslcommerzClient()`** — accepts a custom `fetch` function, which Convex actions support natively
2. **All HTTP transport logic** — timeout handling, AbortSignal, URLSearchParams encoding, error handling
3. **All Zod validation schemas** — re-exported from validators
4. **All TypeScript types** — `CreateSessionRequest`, `CreateSessionResponse`, `IpnPayload`, etc.
5. **Error classes** — `SslcommerzError`, `SslcommerzHttpError`, `SslcommerzValidationError`

**How it works in a Convex action:**

```typescript
// Inside a Convex action (runs in Node.js-compatible V8 runtime)
import { createSslcommerzClient } from "better-sslcommerz";

const client = createSslcommerzClient({
  storeId: process.env.SSLCOMMERZ_STORE_ID!,
  storePasswd: process.env.SSLCOMMERZ_STORE_PASSWD!,
  environment: "live",
  // Convex actions have native fetch — no need to pass custom fetch
});

const session = await client.core.createSession({ ... });
```

**Dependency in the Convex component's package.json:**

```json
{
  "dependencies": {
    "better-sslcommerz": "^0.1.0" // once published
  }
}
```

**Work savings:**

- No need to re-implement HTTP transport
- No need to re-implement Zod schemas
- No need to re-implement TypeScript types
- The validators package does NOT need to be published separately — `better-sslcommerz` already re-exports everything

**Limitations / Caveats:**

1. `better-sslcommerz` is currently version `0.0.0`. It must be published to npm BEFORE building the Convex component.
2. The Convex component cannot import from the local workspace package `better-sslcommerz` directly (the component will be its own standalone project, not part of the monorepo when consumed by users). It must reference the published npm package.
3. During development, you can use `file:../../packages/better-sslcommerz` or `npm link` for local testing.

**Verdict: Reuse `better-sslcommerz` as a dependency. It eliminates rewriting ~500 lines of HTTP/validation code.**

### Track B: Reuse `@better-sslcommerz/validators` Directly

**Feasibility: NO — not as workspace dependency**

The validators package is `"private": true` and lives in the monorepo. The Convex component will be published as a separate standalone npm package. It cannot declare `@better-sslcommerz/validators` as a workspace dependency.

**Options to solve this:**

1. **Publish validators to npm** — Remove `"private": true`, give it a proper version, publish as `@better-sslcommerz/validators`. Then the Convex component can depend on it. This also means the validators package needs a build step that produces a proper dist.
2. **Vendor/copy the schemas** — Copy the validator schemas into the Convex component package under `src/validators/`. Full duplication, but complete isolation.
3. **Use `better-sslcommerz`** — Which already re-exports all validators. This is Track A.

**Verdict: Track B is not needed because Track A (using `better-sslcommerz`) already brings in all the validators.**

### Track C: The Convex-Specific Validators (NEW REQUIREMENT)

Inside the Convex component, we need **Convex validators** (`v.string()`, `v.number()`, etc. from `convex/values`) for the DB schema and function argument validation. These are **separate** from the Zod schemas in the validators package.

The Convex component defines its own DB tables to persist:

- Payment session records (keyed by `tran_id` from SSLCommerz)
- IPN callback records (raw payload + parsed data)
- Transaction status records

These Convex validators are defined inline in `schema.ts` using `v` from `convex/values` — no external dependency needed.

## 7. SSLCommerz-Specific Design Decisions

### 7.1 What Tables Do We Need?

SSLCommerz is **fundamentally different from Stripe/Polar** in its payment flow:

**Stripe flow:** Create checkout session → User completes payment → Stripe sends webhook → App handles webhook
**Polar flow:** Create checkout → User completes → Polar sends webhook → App handles webhook
**SSLCommerz flow:**

1. App creates session → Gets `GatewayPageURL` and `sessionkey`
2. App redirects user to `GatewayPageURL`
3. User completes payment on SSLCommerz gateway
4. SSLCommerz **redirects** user back to `success_url`, `fail_url`, or `cancel_url` with POST data
5. SSLCommerz **also** sends IPN (Instant Payment Notification) to `ipn_url` via POST
6. App should **validate** the order using `validator/api/validationserverAPI.php` before fulfilling

This means we need to persist:

- **Payment sessions** — created when `createSession` is called, updated when redirect/IPN arrives
- **IPN events** — each IPN callback recorded for audit/idempotency
- **Transaction records** — validated/confirmed payments

### 7.2 Proposed Convex Schema

```typescript
// src/component/schema.ts

  paymentSessions: defineTable({
    // Session creation data
    tranId: v.string(), // tran_id — the unique order ID from the app

    gatewayUrl: v.string(), // GatewayPageURL for redirect
    status: v.string(), // "pending" | "success" | "failed" | "cancelled" | "expired"
    amount: v.string(), // total_amount
    currency: v.string(), // BDT, USD, etc.
    environment: v.string(), // "sandbox" | "live"

    // Optional user/org lookup fields
    userId: v.optional(v.string()),
    orgId: v.optional(v.string()),
    metadata: v.optional(v.any()), // custom app data

    // IPN/validation fields (populated after payment)
    valId: v.optional(v.string()), // val_id from IPN
    bankTranId: v.optional(v.string()), // bank_tran_id
    cardType: v.optional(v.string()), // card_type
    storeAmount: v.optional(v.string()), // store_amount (after gateway fee)

    // Timestamps
    expiresAt: v.optional(v.number()), // session expiry
  })
    .index("by_tran_id", ["tranId"])
    .index("by_session_key", ["sessionKey"])
    .index("by_user_id", ["userId"])
    .index("by_org_id", ["orgId"])
    .index("by_status", ["status"]),

  ipnEvents: defineTable({
    // Raw IPN data
    tranId: v.string(),
    valId: v.string(),
    status: v.string(), // "VALID" | "VALIDATED" | "INVALID TRANSACTION" | etc.
    amount: v.string(),
    storeAmount: v.string(),
    currency: v.string(),
    bankTranId: v.optional(v.string()),
    cardType: v.string(),
    cardNo: v.optional(v.string()),
    riskLevel: v.optional(v.union(v.literal("0"), v.literal("1"))),
    riskTitle: v.optional(v.string()),

  })

    .index("by_val_id", ["valId"])
    .index("by_status", ["status"]),
});
```

### 7.3 IPN Handling — The Critical Difference

Unlike Stripe/Polar where there is one standard webhook endpoint, SSLCommerz IPN is:

1. A **POST request** from SSLCommerz to the `ipn_url` you provided when creating the session
2. Contains form-urlencoded data (not JSON)
3. Should be **validated** against the validation API before trusting
4. Does NOT use HMAC signature verification like Stripe webhooks — instead you validate by calling back to SSLCommerz's validation endpoint

The IPN HTTP action must:

1. Parse the `application/x-www-form-urlencoded` body
2. Validate the IPN payload using `ipnPayloadSchema`
3. Store the raw IPN event in the `ipnEvents` table (idempotent — check `val_id`)
4. Call `client.core.validateOrder({ val_id })` to confirm with SSLCommerz API
5. Update the `paymentSessions` table with final status
6. Call the app's custom callback if provided

### 7.4 Success/Fail/Cancel URL Handling

SSLCommerz also POSTs to the success/fail/cancel URLs with payment data. These are NOT webhook-style — the user's browser is redirected there. The app needs to:

1. Parse the POST body on the success/fail/cancel URL handler
2. Either use an HTTP action in Convex for these URLs too, OR handle in the app's web server

For maximum flexibility, we should provide separate HTTP action handlers for `success`, `fail`, and `cancel` redirects — mirroring how Stripe's `registerRoutes` works but with three separate paths.

### 8.5 `src/client/index.ts`

The main `SslCommerzConvex` class:

```typescript
import type { HttpRouter } from "convex/server";
import { createSslcommerzClient } from "better-sslcommerz";
import { httpActionGeneric } from "convex/server";

import type { ComponentApi } from "../component/_generated/component.js";

export class SslCommerzConvex {
  private _storeId: string;
  private _storePasswd: string;
  private _environment: "sandbox" | "live";

  constructor(
    public component: ComponentApi,
    private config?: {
      storeId?: string;
      storePasswd?: string;
      environment?: "sandbox" | "live";
    },
  ) {
    this._storeId = config?.storeId ?? process.env.SSLCOMMERZ_STORE_ID ?? "";
    this._storePasswd =
      config?.storePasswd ?? process.env.SSLCOMMERZ_STORE_PASSWD ?? "";
    this._environment =
      config?.environment ??
      (process.env.SSLCOMMERZ_ENVIRONMENT as "sandbox" | "live") ??
      "sandbox";
  }

  private get sslClient() {
    // Creates a new client per call (stateless)
    return createSslcommerzClient({
      storeId: this._storeId,
      storePasswd: this._storePasswd,
      environment: this._environment,
    });
  }

  async createPaymentSession(ctx: ActionCtx, args: CreatePaymentSessionArgs) {
    const client = this.sslClient;
    const response = await client.core.createSession({
      tran_id: args.tranId,
      total_amount: args.totalAmount,
      currency: args.currency,
      success_url: args.successUrl,
      fail_url: args.failUrl,
      cancel_url: args.cancelUrl,
      ipn_url: args.ipnUrl,
      cus_name: args.customerInfo.name,
      // ... map all fields
    });

    if (response.status !== "SUCCESS") {
      throw new SslCommerzError(
        `Session creation failed: ${response.failedreason}`,
      );
    }

    // Store the session record in Convex
    await ctx.runMutation(this.component.public.createPaymentSessionRecord, {
      tranId: args.tranId,
      sessionKey: response.sessionkey!,
      gatewayUrl: response.GatewayPageURL!,
      amount: String(args.totalAmount),
      currency: args.currency,
      environment: this._environment,
      userId: args.userId,
      orgId: args.orgId,
      metadata: args.metadata,
    });

    return {
      sessionKey: response.sessionkey!,
      gatewayUrl: response.GatewayPageURL!,
      tranId: args.tranId,
    };
  }

  getPaymentSession(ctx: RunQueryCtx, args: { tranId: string }) {
    return ctx.runQuery(this.component.public.getPaymentSession, args);
  }

  listPaymentSessionsByUserId(ctx: RunQueryCtx, args: { userId: string }) {
    return ctx.runQuery(
      this.component.public.listPaymentSessionsByUserId,
      args,
    );
  }

  async validatePayment(ctx: ActionCtx, args: { valId: string }) {
    const client = this.sslClient;
    const result = await client.core.validateOrder({ val_id: args.valId });
    // Update the session status based on validation result
    if (result.status === "VALID" || result.status === "VALIDATED") {
      await ctx.runMutation(this.component.public.updatePaymentSessionStatus, {
        tranId: result.tran_id,
        status: "success",
        valId: result.val_id,
        bankTranId: result.bank_tran_id,
        cardType: result.card_type,
        storeAmount: result.store_amount,
      });
    }
    return result;
  }

  registerRoutes(http: HttpRouter, config?: RegisterRoutesConfig) {
    const ipnPath = config?.ipnPath ?? "/sslcommerz/ipn";
    const successPath = config?.successPath ?? "/sslcommerz/success";
    const failPath = config?.failPath ?? "/sslcommerz/fail";
    const cancelPath = config?.cancelPath ?? "/sslcommerz/cancel";

    http.route({
      path: ipnPath,
      method: "POST",
      handler: httpActionGeneric(async (ctx, req) => {
        // Parse form-urlencoded IPN body
        const body = await req.text();
        const params = new URLSearchParams(body);
        const raw = Object.fromEntries(params.entries());

        // Validate against Zod schema
        const parsed = ipnPayloadSchema.safeParse(raw);
        if (!parsed.success) {
          return new Response("Invalid IPN payload", { status: 400 });
        }

        const ipn = parsed.data;

        // Store IPN event (idempotent via val_id)
        await ctx.runMutation(this.component.internal.recordIpnEvent, {
          tranId: ipn.tran_id,
          valId: ipn.val_id,
          status: ipn.status,
          amount: ipn.amount,
          storeAmount: ipn.store_amount,
          currency: ipn.currency,
          bankTranId: ipn.bank_tran_id,
          cardType: ipn.card_type,
          cardNo: ipn.card_no,
          rawPayload: raw,
          verified: false,
        });

        // Validate with SSLCommerz API
        try {
          const client = this.sslClient; // storeId/passwd from env
          const validation = await client.core.validateOrder({
            val_id: ipn.val_id,
          });

          if (
            validation.status === "VALID" ||
            validation.status === "VALIDATED"
          ) {
            await ctx.runMutation(this.component.internal.markIpnVerified, {
              valId: ipn.val_id,
            });
            await ctx.runMutation(
              this.component.public.updatePaymentSessionStatus,
              {
                tranId: ipn.tran_id,
                status: "success",
                valId: ipn.val_id,
                bankTranId: validation.bank_tran_id,
                cardType: validation.card_type,
                storeAmount: validation.store_amount,
              },
            );

            // Custom user callback
            if (config?.onIpn) {
              const session = await ctx.runQuery(
                this.component.public.getPaymentSession,
                { tranId: ipn.tran_id },
              );
              await config.onIpn(ctx, ipn, session);
            }
          }
        } catch (err) {
          console.error("IPN validation failed:", err);
        }

        return new Response("OK", { status: 200 });
      }),
    });

    // Success redirect handler
    http.route({
      path: successPath,
      method: "POST",
      handler: httpActionGeneric(async (ctx, req) => {
        const body = await req.text();
        const params = new URLSearchParams(body);
        const raw = Object.fromEntries(params.entries());
        const tranId = raw.tran_id;
        const valId = raw.val_id;

        if (tranId && valId) {
          await ctx.runMutation(
            this.component.public.updatePaymentSessionStatus,
            {
              tranId,
              status: "success",
              valId,
            },
          );
          if (config?.onSuccess) {
            const session = await ctx.runQuery(
              this.component.public.getPaymentSession,
              { tranId },
            );
            await config.onSuccess(ctx, session);
          }
        }

        // Redirect to the app's success page
        const redirectUrl = config?.successRedirectUrl ?? "/";
        return Response.redirect(redirectUrl, 302);
      }),
    });

    // Fail and cancel handlers follow same pattern...
  }
}

export function registerRoutes(
  http: HttpRouter,
  component: ComponentApi,
  config?: RegisterRoutesConfig,
) {

  // For apps that don't want to instantiate SslCommerzConvex just for routes
}
```

### 10.3 Key TypeScript differences from current `better-sslcommerz` package

| Aspect            | Current `better-sslcommerz`       | New Convex Component                                                |
| ----------------- | --------------------------------- | ------------------------------------------------------------------- |
| Bundler           | `tsdown` (bundles to single file) | `tsc` only (preserves module structure)                             |
| Output formats    | ESM + CJS                         | ESM only                                                            |
| rootDir structure | `src/` → `dist/` flat             | `src/component/`, `src/client/`, `src/react/` → mirrored in `dist/` |
| DTS               | Separate tsc step after tsdown    | Inline with tsc (`declaration: true`)                               |
| moduleResolution  | Bundler                           | Bundler (same)                                                      |

The key reason for NOT using tsdown/bundler: Convex components must preserve their module graph. The generated `_generated/component.ts` file uses specific relative imports that a bundler would collapse. Convex's own tooling processes the `src/component/` TypeScript files directly during `convex dev`.

### 13.2 `pnpm-workspace.yaml` Addition

The new package is already covered if `pnpm-workspace.yaml` includes `packages/*`. No change needed if that pattern exists.

### 13.3 Turbo Pipeline Configuration

In `turbo.json`, the new package participates in the same `build`, `dev`, `typecheck`, `lint` pipeline as other packages. However:

- The `build:codegen` step (convex codegen) is NEW and not in the current turbo pipeline
- This needs a separate script or pre-build step
- The `_generated/` directory output should NOT be cached by turbo (it must always regenerate)

Recommended approach: make `build:codegen` part of the `build` script sequence with `predev` guard (same as stripe/polar).

### 13.4 `.gitignore` Additions for the New Package

In `packages/convex-better-sslcommerz/.gitignore`:

```

node_modules/
dist/
*.tsbuildinfo
.env.local
```

The `_generated/` directory inside the component IS committed because it contains the `ComponentApi` type that is published as part of the package and needed by TypeScript consumers.

### 13.5 Dependency Chain


Since the Convex component depends on `better-sslcommerz`, they need coordinated versioning. Recommendation:
